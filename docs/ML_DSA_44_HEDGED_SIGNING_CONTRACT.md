# ML-DSA-44 Hedged-Signing Contract

## Status: ENGINEERING_CONTRACT - RELEASE_HOLD
## Spec-ID: ML-DSA-44-HEDGED-SIGNING-v1
## Updated: 2026-07-19
## Consensus-Relevant: NO

## Decision

Any future PQBTC ML-DSA-44 production signer must expose hedged signing only.
It must obtain a fresh 32-byte `rnd` inside the same cryptographic-module
boundary that invokes `ML-DSA.Sign_internal`. Randomness failure is a signing
failure: no deterministic, all-zero, stale, caller-supplied, or partial output
may be returned.

This document is a project-owned engineering contract and partial remediation
for issue `#184`. The executable model in
`contrib/ml-dsa-engineering/hedged_signing_contract.py` makes the error and
concurrency behavior testable, but it is not a cryptographic implementation.
It is not linked into the node, wallet, Script, or an `ALG_ID`, and it does not
close issue `#184` or remove `RELEASE_HOLD`.

`ML_DSA_44_BACKEND_ADMISSION.md` admits the pinned `mldsa-native` portable-C
path for a separate isolated wrapper prototype. That decision supplies a
bounded implementation target for this contract, not a production backend.
The production backend remains `NONE`, and all platform, lifecycle, review,
and release gates remain open.

## Standards Basis

FIPS 204 makes hedged signing the default and permits deterministic signing as
an optional variant. Sections 3.3, 5.2, and 5.4 establish the boundaries
relevant to PQBTC:

- hedged signing obtains a fresh 256-bit value `rnd` from a random bit
  generator;
- signing returns an error if random bit generation fails;
- `rnd` must be generated by the cryptographic module that runs
  `ML-DSA.Sign_internal`;
- implementing only hedged signing is sufficient for interoperability; and
- fresh randomness helps mitigate side-channel and fault attacks, while the
  deterministic variant is harder to protect on exposed platforms.

FIPS 204 Section 3.6.1 requires an ML-DSA-44 key-generation RBG to have at
least 128 bits of security and recommends at least 192 bits to retain the
profile's category-2 claim. That key-generation requirement is separate from
the per-signature `rnd` requirement. PQBTC nevertheless requires the future
signing source to be an operating-system or hardware RBG suitable for
cryptographic key material; it does not accept a project-owned userspace DRBG
without a separate design, validation, reseed, fork, and failure review.

Controlling source:
https://csrc.nist.gov/pubs/fips/204/final

## Public Interface

The production boundary has one signing mode:

```text
Sign(signing_key_handle, verification_key, message, context)
    -> signature | HedgedSignError
```

The interface must not accept `rnd`, an entropy callback, a deterministic-mode
flag, a seed, or a test vector. Fixed-`rnd` and deterministic operations may
exist only in separately built ACVP/comparator tools. They must not be linked
into, exported from, or dynamically reachable through a production binary.

The inputs and successful output are exact:

| Field | Contract |
| --- | --- |
| ML-DSA mode | FIPS 204 external, pure `ML-DSA-44` |
| Message | byte string supplied by the later protocol binding |
| Context | 0 through 255 bytes; the later protocol must freeze its value |
| `rnd` | exactly 32 fresh bytes, generated internally |
| Verification key | exact canonical 1,312-byte key in a production binding |
| Signature | exact canonical 2,420-byte signature |

The executable model keeps key types opaque because wallet storage and backend
selection remain open findings. It rejects missing key handles, non-byte
message/context values, and contexts longer than 255 bytes before requesting
randomness.

The model's private backend protocol is an executable test seam, not a proposed
production FFI. In a real binding, entropy acquisition and
`ML-DSA.Sign_internal` must reside within one reviewed cryptographic-module
boundary. Passing `rnd` across an opaque provider, process, device, or FFI
boundary does not satisfy this contract unless that complete boundary is
specified and shown to meet FIPS 204.

For the admitted isolated prototype, the intended module is one portable-C
translation unit containing the project wrapper, project random-byte and
zeroization hooks, and the pinned `mldsa-native` source. All upstream entry
points are `static`; only the project hedged-signing wrapper and strict
verification surface may be exported. A separately named test build may expose
fixed-randomizer operations for official vectors, but those symbols may not
appear in the production-shaped prototype.

## Entropy Contract

The general-host production design must request fresh bytes directly from the
supported operating system's cryptographic RBG for every signature. It must
not cache a userspace random pool, silently retry through a weaker provider,
or substitute deterministic output. A hardware signer must generate `rnd`
inside the device boundary; the host may request a signature but may not inject
or select the randomizer.

Each supported adapter must document an RBG security strength of at least 128
bits; a 192-bit or stronger source is preferred alongside the ML-DSA-44
category-2 posture. The operating system owns entropy collection, health
handling, and reseeding. PQBTC calls it for every signature and maintains no
project DRBG to seed or reseed. A platform whose RBG strength, reseed behavior,
fork/clone behavior, or failure reporting cannot be reviewed is not an approved
signing platform. A hardware signer's internal RBG and reseed policy require
the same platform-specific review.

The source operation is all-or-nothing. The signer accepts exactly 32 bytes and
fails closed on an exception, unavailable provider, short or malformed result,
or provider-declared failure. PQBTC additionally rejects an all-zero result and
an immediately repeated result as deterministic-downgrade and catastrophic-
source guards. Those checks are stricter project policy, not FIPS RBG health
tests, and they do not certify source quality.

The repeat guard stores only a digest of the last accepted randomizer. The
comparison and update are atomic. A collision or false rejection has negligible
probability under the required source, but the guard detects only consecutive
reuse visible to the same signer state.

## Failure Contract

Every failure returns an error and no signature:

| Error | Required behavior |
| --- | --- |
| invalid input | do not request randomness or invoke the backend |
| entropy source failure | do not invoke the backend; no fallback |
| short or malformed entropy result | do not invoke the backend; no fallback |
| all-zero entropy result | do not invoke the backend; no fallback |
| immediately repeated entropy result | do not invoke the backend; no fallback |
| backend signing failure | discard every partial result |
| wrong signature type or length | do not verify or return it |
| verification error or non-acceptance | return no signature |

The boundary must self-verify every generated signature with the corresponding
verification key before release. Only an exact success result is accepted.
This is one fault-control layer, not a complete fault model; issue `#186`
remains open.

The implementation must destroy `rnd`, expanded secret-key material, sampling
state, and partial signatures on every exit path using compiler-resistant
facilities appropriate to the selected language and backend. The Python model
overwrites its mutable `rnd` buffer so cleanup paths are executable, but Python
cannot establish compiler-resistant erasure. Issue `#187` remains open.

## Concurrency And Lifecycle

Calls through one signer instance are serialized across entropy acquisition,
backend signing, self-verification, cleanup, and result release. This avoids
concurrent use of mutable backend or key state and makes the immediate-repeat
guard atomic. A future implementation may relax serialization only with a
reviewed backend/thread-safety contract and equivalent randomizer guarantees.

The lifecycle boundary is explicit:

| Event | Required behavior and residual boundary |
| --- | --- |
| thread concurrency | serialize one signer instance; never release two operations using one repeated `rnd` |
| POSIX fork | do not carry a project userspace DRBG; child and parent request fresh bytes from the OS before signing |
| process restart | request fresh OS randomness; in-memory repeat history is not persistent |
| clone after an accepted `rnd` | copied guard state rejects that same immediately repeated value |
| clone before first use | no in-memory guard can detect identical future RBG output across both clones |
| VM snapshot/rollback | in-guest state can roll back together; require a platform RBG/hardware boundary whose safety survives the supported snapshot model |
| hardware signer | device generates `rnd`, owns health/failure behavior, and returns no partial signature |

Snapshot rollback before entropy acquisition cannot be solved by an
application-only in-memory guard: the guard and any local DRBG state can roll
back together. A platform that cannot provide fresh post-clone randomness is
not an approved signing platform. Hardware or external monotonic state would
need a separate protocol and threat review before changing that boundary.

## Executable Evidence

Run the bounded contract tests with:

```bash
python3 -m unittest ci.test.test_ml_dsa_hedged_signing_contract
```

They cover:

- a hedged-only public API with no caller-supplied randomizer;
- unavailable, failed, short, all-zero, and repeated entropy;
- immediate-repeat detection after a copied-state snapshot;
- atomic concurrent callers and serialized backend use;
- backend error, wrong-length output, verification rejection, and verification
  error;
- cleanup of the model's mutable randomizer buffer; and
- invalid input rejection before entropy acquisition.

## Exit Criteria

The isolated prototype may bind this contract only under
`ML_DSA_44_BACKEND_ADMISSION.md`. A production proposal still must:

1. re-pin a then-current tagged backend, freeze the imported source and build,
   and define supported platforms and secret ownership types;
2. keep entropy generation and `ML-DSA.Sign_internal` in one reviewed module
   and implement the production API without linking test/ACVP entry points;
3. use reviewed OS and hardware RBG adapters with injected failure coverage;
4. demonstrate compiler-resistant cleanup across language and FFI boundaries;
5. add fork/process and supported VM-clone tests at the platform layer;
6. pass side-channel, fault, fuzzing, advisory, and lifecycle gates; and
7. receive re-review under issue `#181` against the exact implementation
   commit.

Until then, this is a misuse-resistant design target only. It authorizes no
production dependency, consensus design, or release-hold change.
